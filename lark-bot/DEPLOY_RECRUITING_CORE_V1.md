# Deploy Recruiting Core V1

This release upgrades the existing Railway service in place. It does not delete
jobs, candidates, applications, channel history, or bot data.

## Before deployment

1. Create a PostgreSQL backup/snapshot and verify that it is readable.
2. Keep the current successful Railway deployment available for code rollback.
3. Upload the **complete release package**. Do not mix selected files from an
   older V14/V16/V19 package with this one.
4. Preserve all existing Railway variables. In particular do not expose or
   replace `DATABASE_URL`, Lark secrets, signing keys, worker tokens, or admin
   credentials.

## Expected startup

The deployment log must include:

```text
[db] schema ready: 20260722_recruiting_core_v1
```

The service root must return `ok`. A checksum conflict or migration exception is
a stop condition: do not delete tables and do not retry manual SQL. Retain the
log and restore the previous code deployment while investigating.

## Manager verification

1. Open the panel and confirm existing jobs and Channel Analytics history are
   still present.
2. Confirm an Open requisition can accept a test row and a Closed requisition
   cannot accept a new row.
3. Generate a new Excel artifact. Confirm its Job dropdown contains current
   Open jobs and that an existing row on a now-closed job remains readable.
4. Submit one test row, then submit the identical artifact again. The second
   submission must add no duplicate candidate, application, or stage event.
5. Change the same application in the website or Lark, then try the older Excel
   row. It must report a version conflict rather than overwrite newer work.
6. In Lark, change one valid existing application's stage and submit. Refresh
   `/channel_sheet`; the snapshot should show the latest successful submission.
7. Confirm Channel Analytics counts baseline imports separately from true new
   intake and derives later funnel movement from stage events.

## Safe rollback

The migration is additive and recorded once. If application code must be rolled
back, roll back the Railway deployment only. Do not remove migration rows or
new columns. A future fix must use a new migration version rather than editing
the SQL under `20260722_recruiting_core_v1`.

## Artifact policy

- Candidate uploads are XLSX only; CSV is not accepted.
- Only artifacts generated by this service are accepted.
- Maximum compressed XLSX size is 16 MB, with stricter internal package limits.
- A macro, external link, signature failure, duplicate system token, unknown
  job, immutable-field change, or stale version is rejected fail-closed.
